5 # What is this library?
6 
7 This library provides support for Unicode Text, as specified by the
8 Unicode Standard.
9 
10 These are the problems that this library intends to solve (links for examples):
11 
12  * Iterating and truncating on correct boundaries for:

14  * graphemes - [iterate](grapheme_iteration.html), [truncate](truncate_on_grapheme.html).
15  * words
16  * sentences
17  * Normalization

29 # Why?
30 
31 Properly supporting international text is one of these problems that
32 you either think is very easy because you never had to deal with
33 internationalized text, or you give up on it because you think no one
34 can support Unicode correctly. This library is an attempt to provide a
35 simple reusable set of interfaces to deal with international text.
36 
37 # Why not just use the icu library?
38 
39 The first reason for me to consider this was that the icu library is
40 much more complex than what's required for supporting text. ICU also
41 supports the full localization definitions of the Unicode CLDR, which
42 makes its support surface much larger than would be required for
43 simply handling text.
44 
45 Secondly, the icu library forces the internal representation of the
46 text to always be in a particular encoding (UTF16). I believe that
47 this is an unecessary constraint, and one of the points of this
48 library is to make the encoding pluggable without requiring different
49 APIs or explicit conversions.

51 # Some important conversations
52 
53 ## A note on dynamic encodings
54 
55 While it is technically possible to operate on text with a dynamically
56 assigned encoding, I honestly believe the use case for that is more
57 academic than anything else.
58 
59 Your application should have one internal encoding and all text should
60 be converted on the borders of the application to whatever makes the
61 most sense for your application.
62 
63 For instance, if you're dealing with mostly 7-bit ASCII text with the
64 ocasional exception (such as most latin languages), UTF8 is definitely
65 the most efficient usage of memory, and will promote the best cache
66 locality.
67 
68 If, on the other hand, the application is expected to deal with mostly
69 non-latin text, UTF16 is believed to result in the smallest memory
70 usage.
71 
72 But if you don't quite care about memory usage and you have a stronger
73 requirement for being able to do more efficient random access on the
74 text, UTF32 with native endianess will make the most sense.
75 
76 It is, of course, technically feasible to have encodings that make
77 decisions at run time, but the cost of that runtime decision will be
78 paid over and over in the life-cycle of that object.
79 
80 For that reason, it is much more intelligent to convert the text to
81 one of UTF8, UTF16 with native endianess or UTF32 with native
82 endianess at the border of your application. I cannot see a rason to
83 support any other encoding as the basis for operating on the text, any
84 other encoding may be considered a serialization format to be
85 converted to and from in the borders of your application.
86 
87 Note that since the encoding is pluggable, it would be possible for a
88 third-party to contribute support for dynamic encodings.
89 
90 ## A note on endianess
91 
92 The unicode standard defines UTF16BE and UTF16LE, as well as UTF32BE
93 and UTF32LE. The bare "UTF16" and "UTF32" encodings are actually
94 runtime-defined according to Byte-Order-Marker (BOM), the default, in
95 the absense of a BOM is Big-Endian, accorting to the standard -- utf8
96 is always little endian, and BOM characters in utf8 are not relevant.
97 
98 This library introduces the terms UTF16NE and UTF32NE. Those are not
99 encodings, but rather architecture-specific runtime types that will
100 either be the BE or the LE variants depending on the machine that runs
101 it. Whenever you are using the UTF16NE and UTF32NE types, you are
102 expected to appropriately convert the incoming texts to the correct
103 native encoding on the borders of your application.
104 
105 ## A note on legacy encodings
106 
107 This library will not support converting to and from legacy encodings
108 as well as operating on them. You are expected to perform the
109 convertions at the borders of your application.
110 
111 Note that since the encoding is pluggable, it would be possible for a
112 third-party to contribute support for native encodings.
113 
114 ## A note on bound checks
115 
116 UTF8 and UTF16 introduce a new possible type of overflow and
117 underflow. If you are trying to iterate forward, the first octet in
118 UTF8 -- or the first 16bit in UTF16 -- may tell you to look for more
119 characters than what fits in the rest of the buffer. Likewise, if
120 you're iterating backwards, the first octet of your utf8 text -- or
121 16bits in utf16 -- may tell you that you need to look further back,
122 underflowing the buffer.
123 
124 There are two possible ways to handle this, the first is to make the
125 bounds check pervasively in the code. This means that the entire API
126 needs to be more complicated because every operation now needs to know
127 where the string ends, not to mention that the additional checks will
128 have a compounding runtime cost.
129 
130 The alternative, which is what this library implements, is to validate
131 the strings in the borders of your application, whenever you receive a
132 text from the outside, you need to check for correctness on the start
133 and end of your text.
134 
135 This library will offer a utility function to verify the correctness
136 of the string, and optionally forcing the replacement of the invalid
137 utf8 and utf16 sequences by replacement characters.

153 # What are the guiding principles?
154 
155 ## No "natural" unit
156 
157 There is no default unit. Most of the confusion around unicode is
158 derived from the misconceptions on how to iterate over text and how to
159 count and measure it. This libray defines and uses the following
160 units:
161 
162 1. native size: This measure the size of the text in its in-memory
163  representation. It is not necessarily a number of bytes, but it's
164  rather a variable size depending on the encoding and on the
165  architecture where the code is running: for instance, UTF8 always
166  uses octets as the encoding unit, however UTF32 may either use a 32
167  bit int, if the endianess is the same, or it may still be forced to
168  use octets if it's the wrong endianess. The size in bytes is an
169  implementation detail that is architecture-specific and can be
170  measured using operators like sizeof().
171 
172 2. codepoint: The unicode standard calls this "character", but the
173  oveloading of this term is definitely one of the main sources of
174  confusion around the unicode standard, therefore, in this library
175  we just do not ever use the term "character", and prefer the more
176  precise term "codepoint", which represents an encoded unicode
177  entity, which is usually represented as U+DEADBEEF.
178 
179 3. grapheme: The grapheme is the unit of rendering, also referenced in
180  the standard as "grapheme cluster" or "user perceived
181  character". It represents a sequence of one or more codepoints that
182  represent a single graphical symbol, previous versions of the
183  unicode standard had a simple rule of "base character" and
184  "combining characters", but more recent versions of the standard
185  support more complex definitions for grapheme cluster boundaries.
186 
187 4. word: A word is a set of graphemes that form a unit, it can be as
188  simple as "two words" for english, but it requires the use of the
189  unicode database for full internationalization support.
190 
191 5. sentence: A collection of words. There is a property of codepoints
192  in the unicode database that specifies if a given character is a
193  sentence separator or not.

203 # The different aspects of Text
204 
205 In order to allow maximum reusability of the code, each aspect of the
206 text is going to be expressed in this library as a concept. Different
207 traits will implement that concept in different ways in order to at
208 the same time allow room for optimizations. For instance, UTF32 with
209 native endianess will be able to use the native integer type, while
210 UTF32 with the foreign endianess will need to use four octets and will
211 need to do the byte-swapping at runtime for most operations.
212 
213 The usage of traits for this also gives us transparent
214 type-safety. Such that we can statically evaluate if you are
215 incorrectly mixing texts of different encodings, which is the most
216 common form of mistake when handling internationalized text.
217 
218 ## Encoding
219 
220 The encoding concept will provide that basic interface from the
221 in-memory representation to iteration and access of codepoints.
222 
223 ## Normalization Form
224 
225 Normalization is what allows equivalent sequences of characters to be
226 treated as being the same. By making the normalization a trait of the
227 text, we allow the implicit re-normalization when dealing with
228 denormalized text, at te same time that it gives room for important
229 optimizations when two different texts have the same normalization
230 form.
231 
232 ## Locale tailoring
233 
234 The unicode standard has several features that are coupled with the
235 tailoring to specific locales, this will also be applicable to
236 different features. Making this into a trait allows to selectively
237 disable the tailoring for application where the speed gain from not
238 supporting this is more important than the use cases where such
239 support is a requirement. Making this into a trait even allows
240 optimizing away the storage required for storing the language of the
241 text, or if an application needs optimization for the tailoring for a
242 single language, it could be implemented directly as code and avoid
243 any runtime language queries.
